## 5. P0 问题清单

### P0-01 配置导出后无法被当前版本稳定导入

- **类型**：功能缺陷 / 业务闭环
- **模块**：设置页、配置导入导出
- **当前状态**：已完成
- **问题描述**：
  导出 JSON 与导入 JSON 使用的字段契约不一致，连接、任务、设置字段混用了 PascalCase 和 snake_case，`schedule_config`、`task_config` 的结构也不统一。
- **业务影响**：
  现场做配置备份和灾备恢复时，导出的文件未必能重新导回当前版本，影响恢复效率与可靠性。
- **建议修复**：
  统一导入导出 DTO 契约；明确一套标准命名；旧格式如需兼容，做兼容读取，不要混写混读。
- **主要证据**：
  - `src/DBKeeper.App/Views/SettingsPage.xaml.cs`
  - `docs/PRD.md`
- **修复结果**：
  - 新增 `ConfigurationTransferService`，统一导出为稳定的 snake_case JSON 契约。
  - 导入侧兼容 PascalCase / snake_case / kebab-case，并兼容 `schedule_config`、`task_config` 为对象、数组或 JSON 字符串。
  - 导入完成后会返回新增/更新/跳过/缺密码连接等结果，并对启用任务重新挂载调度。
- **验证结果**：
  - 已新增回归测试：导出当前配置后再导入新库，连接、任务、settings 均可回灌成功。
  - `dotnet test` 已通过，覆盖 `ExportAndImportAsync_ShouldRoundTripCurrentContract` 与旧格式兼容场景。
- **最后更新**：
  - 2026-06-23

### P0-02 导入设置时布尔/数字值会抛异常

- **类型**：功能缺陷
- **模块**：设置页、配置导入
- **当前状态**：已完成
- **问题描述**：
  导入 settings 时直接对 JSON 值调用 `GetString()`，当值为 `true/false/90/3` 这类布尔或数字时会抛异常。
- **业务影响**：
  导入可能中途失败，形成“连接导了一半、任务导了一半”的半成功状态。
- **建议修复**：
  对 JSON 值按 `ValueKind` 分类处理，统一转换为字符串存储；导入失败时应回滚或至少中止前做预检。
- **主要证据**：
  - `src/DBKeeper.App/Views/SettingsPage.xaml.cs`
  - `docs/PRD.md`
- **修复结果**：
  - 导入 settings 时已按 `JsonValueKind` 分类处理，布尔、数字、字符串都会按真实值转换后落库。
  - 导入流程已放入 SQLite 事务中，避免中途异常后留下半成功状态。
- **验证结果**：
  - 已新增回归测试：`max_concurrent_tasks=7`、`auto_start=true` 可从 typed JSON 正常导入。
  - `dotnet test` 已通过，未再出现 `GetString()` 类型异常。
- **最后更新**：
  - 2026-06-23

### P0-03 删除备份时磁盘删失败仍物理删记录

- **类型**：功能缺陷 / 审计断裂
- **模块**：备份管理
- **当前状态**：待验证
- **问题描述**：
  删除磁盘文件失败时，系统仍继续把 `backup_files` 记录物理删除，导致界面和数据库都看不到该文件，但磁盘上可能仍然存在。
- **业务影响**：
  后续无法追踪“删没删掉、谁删的、什么时候删的”，不利于日常运维和排障。
- **建议修复**：
  删除成功后只更新为 `DELETED` 状态并记录 `deleted_at`；删除失败时保留记录并展示失败原因。
- **主要证据**：
  - `src/DBKeeper.App/ViewModels/BackupFilesViewModel.cs`
  - `src/DBKeeper.Data/Repositories/BackupFileRepository.cs`
- **修复结果**：
  - 删除逻辑已改为“先删磁盘、成功后把记录更新为 `DELETED` 状态”，不再物理删除数据库记录。
  - 单个删除和批量删除均会汇总失败项，并在界面弹窗提示删除结果。
- **验证结果**：
  - 代码路径、构建与回归测试已通过，未引入编译回归。
  - 仍需现场手工验证“磁盘删除失败时记录仍保留、成功时状态改为 `DELETED`”。
- **最后更新**：
  - 2026-06-23

### P0-04 删除备份缺少路径边界与扩展名校验

- **类型**：业务风险
- **模块**：备份管理
- **当前状态**：待验证
- **问题描述**：
  删除逻辑直接信任 `backup_files.file_path`，没有校验文件是否位于允许的备份目录下，也没有限制扩展名。
- **业务影响**：
  一旦库里路径脏掉，用户在 UI 点“删除备份”时可能删到备份目录外文件。
- **建议修复**：
  删除前校验：
  1. 必须位于已配置备份目录内  
  2. 扩展名只允许 `.bak/.diff/.trn` 等备份文件  
  3. 禁止根目录、系统目录、非备份目录
- **主要证据**：
  - `src/DBKeeper.App/ViewModels/BackupFilesViewModel.cs`
  - `src/DBKeeper.App/Views/BackupFilesPage.xaml.cs`
- **修复结果**：
  - 新增 `BackupPathGuard`，统一校验备份目录边界与允许扩展名，仅允许 `.bak/.diff/.trn`。
  - 备份删除与批量删除前都会经过路径与扩展名检查，避免误删备份目录外文件。
- **验证结果**：
  - 已新增 `BackupPathGuardTests` 覆盖允许目录与扩展名判断。
  - 仍需现场手工验证 UI 删除入口对异常路径的提示是否符合预期。
- **最后更新**：
  - 2026-06-23

### P0-05 保留天数为 0 时会误删历史备份

- **类型**：功能缺陷
- **模块**：清理任务
- **当前状态**：已完成
- **问题描述**：
  `RetentionDays = 0` 时，清理任务按 `DateTime.Now.AddDays(0)` 计算截止时间，实际会把旧文件视为过期文件删除。
- **业务影响**：
  与“0 = 永不删除”的常见业务语义相反，容易造成误删。
- **建议修复**：
  明确约定 `RetentionDays <= 0` 时直接跳过过期删除逻辑，仅保留状态提示。
- **主要证据**：
  - `src/DBKeeper.Executors/CleanupExecutor.cs`
  - `docs/PRD.md`
- **修复结果**：
  - 清理执行器已明确把 `RetentionDays <= 0` 视为“跳过过期删除”，不再按当天时间误判。
- **验证结果**：
  - 已新增 `CleanupExecutorTests` 回归，确认保留天数为 `0` 时不会删除历史文件。
  - `dotnet test` 已通过。
- **最后更新**：
  - 2026-06-23

### P1-01 自定义 SQL 不支持 GO 分隔

- **类型**：业务偏差
- **模块**：自定义 SQL
- **当前状态**：已完成
- **问题描述**：
  PRD 要求多行 SQL 支持 `GO` 分隔，但当前执行器把整段 SQL 直接交给单个 `SqlCommand` 执行。
- **业务影响**：
  现场常见维护脚本无法直接粘贴执行，易造成“界面看起来支持，实际执行失败”。
- **建议修复**：
  在执行前按 `GO` 做批次拆分，并逐段执行；日志里要保留失败批次位置。
- **主要证据**：
  - `src/DBKeeper.Executors/SqlExecutor.cs`
  - `src/DBKeeper.Data/SqlServerClient.cs`
  - `docs/PRD.md`
- **修复结果**：
  - 新增 `SqlBatchSplitter`，按独立 `GO` 行拆分批次。
  - `SqlExecutor` 已改为逐批执行，自定义 SQL 可直接粘贴常见维护脚本。
- **验证结果**：
  - 已新增 `SqlBatchSplitterTests` 回归验证。
  - `dotnet test` 已通过，`GO` 分隔脚本处理正常。
- **最后更新**：
  - 2026-06-23

### P1-02 备份任务中的保留天数未形成自动清理闭环

- **类型**：业务闭环缺失
- **模块**：备份生命周期
- **当前状态**：待验证
- **问题描述**：
  备份任务会写入 `ExpiresAt`，但不会自动按该任务的保留策略清理；真正删文件依赖单独的 `BACKUP_CLEANUP` 任务。
- **业务影响**：
  用户在备份任务里配置了“保留天数”，但系统并不会自动落地执行，容易产生业务误解。
- **建议修复**：
  明确二选一：
  1. 备份任务执行完成后自动带出清理策略  
  2. 界面明确提示“仅记录保留策略，不自动清理，需额外配置清理任务”
- **主要证据**：
  - `src/DBKeeper.Scheduling/SchedulerService.cs`
  - `src/DBKeeper.Executors/CleanupExecutor.cs`
- **修复结果**：
  - 本轮采用“明确提示”方案：在任务编辑框的备份配置中补充说明，明确“保留天数仅记录策略，如需自动删文件需额外配置备份清理任务”。
  - 避免用户误以为单独配置备份任务就会自动删除过期文件。
- **验证结果**：
  - 代码已完成并通过构建回归。
  - 仍需现场手工确认提示文案在任务编辑流程中可见且易理解。
- **最后更新**：
  - 2026-06-23

### P1-03 调度重算时污染最后执行信息

- **类型**：数据准确性
- **模块**：调度器、任务列表
- **当前状态**：已完成
- **问题描述**：

- **业务影响**：
  任务列表里的“上次执行时间/状态”不完全可信，影响排障与值班判断。
- **建议修复**：
  新增只更新 `next_run_at` 的仓储方法，不要复用“更新最后执行状态”的方法。
- **主要证据**：
  - `src/DBKeeper.Scheduling/SchedulerService.cs`
  - `src/DBKeeper.Data/Repositories/TaskRepository.cs`
- **修复结果**：
  - 任务仓储已新增 `UpdateNextRunAsync`，调度器重算下一次执行时间时不再复用 `UpdateLastRunAsync`。
  - `last_run_at` / `last_run_status` 与 `next_run_at` 的写入职责已拆开。
- **验证结果**：
  - 已新增 `SchedulerServiceTests` 回归验证重算仅更新 `next_run_at`。
  - `dotnet test` 已通过。
- **最后更新**：
  - 2026-06-23

### P1-04 备份校验后的文件和状态没有闭环

- **类型**：业务闭环缺失
- **模块**：备份校验、备份登记
- **当前状态**：已完成
- **问题描述**：
  开启“完成后校验”时，若 `VERIFYONLY` 失败，磁盘上可能已经有备份文件，但不会登记进 `backup_files`；即便校验成功，也没有更新 `is_verified`。
- **业务影响**：
  备份文件可能变成 UI 看不到、清理任务也管不到的孤儿文件；同时无法区分哪些备份已校验。
- **建议修复**：
  先登记，再补校验状态；失败时标注异常状态并保留记录，不要让文件脱离管理。
- **主要证据**：
  - `src/DBKeeper.Executors/BackupExecutor.cs`
  - `src/DBKeeper.Scheduling/SchedulerService.cs`
  - `docs/DDD.md`
- **修复结果**：
  - 备份执行器在校验失败但文件已生成时，会把 `BackupCreated`、`IsVerified` 等元数据带回调度层。
  - 调度层会根据元数据登记备份文件并写入 `is_verified`，避免孤儿文件脱管。
- **验证结果**：
  - 已新增 `SchedulerServiceTests` 覆盖“备份已生成但校验失败仍登记文件”的场景。
  - `dotnet test` 已通过。
- **最后更新**：
  - 2026-06-23

- **验证结果**：
  - 已新增 `CleanupExecutorTests`，覆盖允许扩展名与删除状态更新。
  - `dotnet test` 已通过。
- **最后更新**：
  - 2026-06-23

### P1-06 仪表盘磁盘监控不一定反映真实备份盘

- **类型**：体验偏差
- **模块**：仪表盘
- **当前状态**：待验证
- **问题描述**：

### P1-10 导入无事务、无预检、跳过任务不提示

- **类型**：业务恢复可靠性
- **模块**：配置导入
- **当前状态**：待验证

  - 配置导入已改为单事务执行，异常时回滚。
  - 导入结果会返回新增/更新/跳过任务、缺密码连接，并在设置页弹出汇总。
  - 连接已存在时会更新内容，不再一律跳过。
- **验证结果**：
  - 回灌与旧格式兼容测试已通过，导入结果聚合逻辑已接入设置页。
