**生成日期**：2026-04-26  
**文档定位**：后续开发任务的唯一主依据  
**适用范围**：DBKeeper 当前代码、文档整理、后续功能补全与质量改进

---

## 1. 结论摘要

DBKeeper 是面向 SQL Server 的 Windows 单机桌面维护工具，核心目标是帮助 MES 运维工程师或 DBA 完成数据库备份、备份清理、任务调度、执行日志、磁盘监控等日常维护工作。

当前项目已经具备基础框架和部分功能实现，但还不能直接按生产可靠工具标准使用。主要问题不在“功能入口是否存在”，而在以下方面：

- 调度执行可靠性不足，存在重复触发、并发设置不生效、超时不能真正取消 SQL 的风险。
- 备份文件删除和清理逻辑仍有误删、审计断裂、忽略手动保留的问题。
- 文档与代码实际状态严重不一致，旧 TODO、修复任务和审查文档不能继续作为开发依据。
- UI 能完成部分操作，但对运维人员的状态反馈、风险提示、批量操作和排障引导不够。
- 缺少测试工程、CI、发布说明和真实生产部署约束。
- 加密和密码安全不作为后续功能重点；本项目按单机内部工具处理，只保留必要的防误操作、防误删、导入校验、日志追溯能力。

## 2. 当前项目定位

### 2.1 产品定位

DBKeeper 是一个运行在 Windows 服务器上的单机 SQL Server 维护工具，不是通用 SQL 客户端，也不是多用户平台。

核心职责：

- 管理多个 SQL Server 连接。
- 定时执行数据库备份。
- 扫描和管理备份文件。
- 自动清理过期备份。
- 执行存储过程和自定义 SQL。
- 记录任务执行日志。
- 展示任务、连接、磁盘和备份状态。
- 通过托盘维持后台运行。

### 2.2 目标用户

目标用户是 MES 运维工程师、现场维护人员、DBA 或具备 SQL Server 基础操作能力的技术人员。

产品设计应优先考虑：

- 操作直观，不要求写脚本。
- 风险操作必须清楚提示影响范围。
- 出错后能快速定位原因。
- 后台运行时状态可见。
- 备份文件不能误删，关键记录不能丢。

## 4. 当前主要不足

### 4.1 P0：必须优先处理的问题

这些问题会直接影响数据安全、任务可靠性或问题追溯，后续开发应先处理。

| 问题 | 当前表现 | 风险 | 参考位置 |
|---|---|---|---|
| 调度可能重复触发同一任务 | 到期任务未先从调度集合中摘除，长任务期间下一轮 tick 仍可能再次排队 | 同一备份任务并发执行，可能覆盖或损坏备份文件 | `src/DBKeeper.Scheduling/SchedulerService.cs` |
| 并发上限设置未生效 | 设置页可改 `max_concurrent_tasks`，调度器仍固定 `SemaphoreSlim(3, 3)` | 操作人员以为已限流，实际没有生效 | `SchedulerService.cs`、`SettingsPage.xaml.cs` |
| 超时不能真正取消 SQL | 调度层 `WaitAsync` 超时后只放弃等待，底层 SQL 没有取消令牌 | 日志显示失败，但 SQL Server 端可能仍在执行 | `ITaskExecutor.cs`、`SqlServerClient.cs` |
| 连接丢失时不写执行日志 | 找不到连接时直接 return，未创建失败日志 | 任务像“没触发过”，排障困难 | `SchedulerService.cs` |
| 备份删除审计断裂 | 手动删除后直接删除 `backup_files` 记录 | 无法追踪谁删了什么、何时删除 | `BackupFilesViewModel.cs`、`BackupFileRepository.cs` |
| 清理任务忽略手动保留 | `CleanupExecutor` 按磁盘文件直接删，不查询 `is_pinned` | 标记保留的关键备份仍可能被删 | `CleanupExecutor.cs` |
| 导入导出 JSON 不兼容 | 导出字段 PascalCase，导入按小写/下划线读取；任务配置对象/字符串处理不一致 | 灾备配置无法可靠恢复 | `SettingsPage.xaml.cs`、`PRD.md` |
| 文件删除路径缺少边界校验 | 手动删除和清理任务直接信任路径 | 配置错误或数据库脏数据可能导致误删备份外文件 | `BackupFilesViewModel.cs`、`CleanupExecutor.cs` |

### 4.2 P1：影响长期维护的问题

| 问题 | 当前表现 | 建议 |
|---|---|---|
| 缺少 Service/Application 层 | View/ViewModel 直接依赖 Repository/Data，业务散落在 UI、调度和执行器 | 新增应用服务层，ViewModel 只调用服务 |
| SQLite 外键策略与 DDD 不一致 | 连接串关闭外键，删除时把 `task_id`、`connection_id` 写为 0 | 改为明确的软删除或历史快照字段 |
| JSON 契约漂移 | DDD 示例是 snake_case，代码大量使用 PascalCase | 统一采用一种契约，并提供兼容读取 |
| 备份校验结果未落库 | 表有 `is_verified`，执行 `RESTORE VERIFYONLY` 后不更新 | 校验完成后更新备份记录 |
| 启动清理说明与实现不一致 | 注释写清理已删除备份记录，代码只清理执行日志 | 补实现或修正文档 |
| 备份同步服务可能重入 | Timer fire-and-forget，无扫描互斥 | 增加扫描锁，避免并发扫描同一批记录 |
| 数据库列表加载缺少明确反馈 | 获取失败只写日志，用户不知道原因 | 在任务编辑框内显示失败信息和重试入口 |

### 4.3 P2：体验和扩展问题

| 问题 | 当前表现 | 建议 |
|---|---|---|
| 操作反馈不统一 | 多数操作只写日志，没有 Snackbar/InfoBar | 建立统一反馈组件 |
| 连接删除风险提示不完整 | 关联任务影响没有稳定展示 | 删除前列出影响任务，并支持改绑 |
| 默认备份目录手输不保存 | 文本框可编辑，但只有浏览按钮会落库 | 失焦保存或改成只读 |
| 托盘交互不一致 | UI_SPEC 写单击打开，实现是双击 | 统一规格和实现 |
| 任务列表排障不便 | 缺少下次执行、运行中、失败状态筛选 | 补列和过滤器 |
| 备份批量操作弱 | 缺少全选、反选、按异常选择、释放空间预估 | 增加批量工具条 |
| 概览连接状态未绑定心跳 | 视觉点始终中性，未显示真实在线状态 | 接入 `ConnectionHeartbeatService` 状态 |

## 5. 后续开发任务

### 5.1 P0：稳定可用前必须完成

1. **防重复执行**
   - 同一任务同时只能有一个运行实例。
   - 到期任务触发前先原子标记为运行中或从调度集合摘除。
   - 手动执行与定时执行也要共享同一互斥规则。

2. **并发上限真正生效**
   - 调度器启动时读取 `max_concurrent_tasks` 初始化并发锁。
   - 设置变更后要么立即重建调度器并提示，要么提示重启生效。

3. **超时和取消机制**
   - `ITaskExecutor.ExecuteAsync` 增加 `CancellationToken`。
   - SQL 命令传递取消令牌。
   - 超时后日志写明“已请求取消”，不要假装底层一定停止。

4. **连接缺失也记录日志**
   - 无连接、连接配置缺失、密码为空、数据库不可达，都应记录失败日志。
   - 日志摘要要能让操作人员知道应该去哪里修。

5. **删除改为状态流转**
   - 手动删除成功后更新为 `DELETED`，记录 `deleted_at`。
   - 删除失败时不能删除数据库记录。
   - 批量删除要汇总成功、失败、释放空间。

6. **清理任务遵守手动保留**
   - 清理候选应来自 `backup_files` 活跃记录，而不是直接枚举目录全删。
   - `is_pinned = true` 的文件永不自动清理。
   - 清理前校验路径在已知备份目录内。

7. **修复导入导出**
   - 导出的配置必须能被当前版本重新导入。
   - 导入前做预检，展示将新增、跳过、冲突、缺失密码、危险路径。
   - 导入后提示需要补填的连接信息和受影响任务。

8. **路径边界与防误删**
   - 删除文件只允许 `.bak`、`.diff`、`.trn` 等明确备份扩展名。
   - 删除路径必须位于已配置备份目录内。
   - 禁止把磁盘根目录、系统目录作为清理目录。

### 5.2 P1：结构和可维护性

1. **建立应用服务层**
   - 新增任务服务、连接服务、备份文件服务、设置服务、配置导入导出服务。
   - ViewModel 不再直接编排仓储和调度细节。
   - View 只负责弹窗、输入和导航。

2. **统一 JSON 契约**
   - 选择 PascalCase 或 snake_case 作为内部标准。
   - 导入旧配置时兼容另一种命名。
   - 所有 `TaskConfig`、`ScheduleConfig` 读写通过统一 DTO。

3. **修复数据一致性策略**
   - 明确任务删除、连接删除后历史日志和备份记录如何保留。
   - 不再用 `0` 伪外键表达历史数据。
   - 必要时新增快照字段，例如连接名、任务名、数据库名。

4. **备份校验落库**
   - `RESTORE VERIFYONLY` 成功后更新 `is_verified = 1`。
   - 校验失败后写日志并标记备份状态。

5. **统一操作反馈**
   - 成功操作使用 Snackbar。
   - 失败操作使用就近错误提示或 InfoBar。
   - 耗时操作显示进度，按钮禁用防重复点击。

### 5.3 P2：扩展能力

1. **备份恢复向导**
   - 从备份文件列表选择恢复。
   - 恢复前执行校验。
   - 支持恢复到新库名。
   - 覆盖现有库必须二次确认。
   - 恢复过程写执行日志。

2. **备份链管理**
   - 支持全量、差异、日志备份链展示。
   - 标记备份链断裂。
   - 展示最近可恢复点。

3. **任务模板**
   - 每日全备模板。
   - 每周全备 + 每日差异模板。
   - 日志备份模板。
   - 清理任务模板。

4. **运行健康看板**
   - 连续失败任务。
   - 最近 7 天成功/失败趋势。
   - 磁盘空间趋势。
   - 备份增长趋势。
   - 长时间未成功备份的数据库。

5. **测试和发布流程**
   - 新增测试工程。
   - 覆盖调度计算、清理策略、导入导出、备份记录状态流转。
   - 增加 CI 构建和测试。
   - 编写发布包、升级、回滚说明。

## 6. 操作人员体验改进计划

### 6.1 连接管理

- 删除连接前显示影响分析：关联任务数量、任务名称、最近执行状态。
- 如果连接有关联任务，提供“取消删除”“禁用关联任务”“改绑到其他连接”。
- 测试连接时显示进度和结果，不只改变卡片文本。
- 心跳状态需要在概览和侧边栏都能看到。

### 6.2 任务管理

- 任务列表增加下次执行时间、距离执行剩余时间、上次状态、运行中状态。
- 立即执行前提示任务类型、目标数据库、预计风险。
- 运行中的任务禁止重复点击立即执行。
- 编辑任务时，如果数据库列表加载失败，显示原因和重试按钮。

### 6.3 备份管理

- 批量删除前展示文件数量、总大小、最早/最新备份时间。
- 删除失败时保留记录并显示失败原因。
- 手动保留状态在列表中醒目展示。
- 增加“仅异常”“仅过期”“仅未校验”“仅未保留”等快捷筛选。

### 6.4 设置与导入导出

- 默认备份目录要么只读，要么手动输入后失焦保存。
- 导入配置改为三步：选择文件、预检结果、确认导入。
- 导入完成后列出需要补填密码或测试连接的项目。
- 清空日志、批量删除、恢复数据库等高风险操作统一二次确认。

### 6.5 托盘和后台运行

- 明确单击或双击打开主窗口，文档和实现保持一致。
- 托盘菜单显示当前状态：运行中、已暂停、失败任务数。
- 暂停所有任务后主窗口也应显示明显状态。

## 9. 后续执行原则

- 先修会造成数据丢失、重复执行、无法追溯的问题，再做新功能。
- 不再依据旧 TODO 和旧修复任务文档开发。
- 所有删除、清理、恢复、导入操作都必须先做影响分析。
- 文档、代码、界面三者必须同步更新。
- 后续每完成一组任务，应更新本文的状态或拆出新的实施清单。
